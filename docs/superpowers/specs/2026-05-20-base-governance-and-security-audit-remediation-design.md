# Base Governance And Security Audit Remediation Design

## 1. 目标

本轮设计处理 `pantheon-base` 标准后台里 6 组已确认问题：

1. 右侧功能栏分页缺少“回到第一页 / 跳到最后一页”能力
2. `system/org` 的部门管理首屏加载卡顿，岗位管理治理头部缺失明确子域
3. `system/config` 的字典管理治理头部缺失明确子域
4. `system/config` 的国际化管理首屏加载卡顿
5. `platform/lowcode` 的模块管理治理头部缺失子域，模块生成器缺少治理信息
6. `system/auth + system/audit` 的登录日志、会话管理、操作日志缺少完整安全事件闭环

本轮以 `pantheon-base` 为唯一主修复仓库。`pantheon-ops` 不在本轮直接落代码，只输出后续继承同步清单。

---

## 2. 设计结论

### 2.1 平台通用能力

- 右侧功能栏分页增强为平台级通用组件能力
- 新增首页和末页跳转，不允许在单页单表里分别拼接一套按钮逻辑

### 2.2 治理头部语义

- 岗位管理归属 `system/org`
- 字典管理归属 `system/config`
- 模块管理和模块生成器归属 `platform/lowcode`
- 所有治理页顶部必须明确显示主域与子域，不再出现只有“治理”但没有域语义的头部

### 2.3 卡顿页处理策略

- 部门管理与国际化管理都按“首屏链路拆分 + 后端查询收敛”处理
- 不接受只增加骨架屏、延长 loading 或隐藏慢查询的表层修补

### 2.4 安全审计处理策略

- 登录日志归 `system/auth`
- 会话管理归 `system/auth`
- 操作日志归 `system/audit`
- 安全事件作为跨 `auth/audit` 的闭环对象补齐最小完整生命周期

### 2.5 数据与接口策略

- 若旧设计阻塞完整闭环，允许做中等力度重构与迁移
- 不局限于前端补按钮，可新增字段、索引、表、DTO、服务和权限点

## 3. 范围边界

### 3.1 主层：`platform`

`platform` 负责：

- 右侧功能栏分页契约
- 通用治理页头部元数据契约
- `platform/lowcode` 的治理语义补齐

它不负责：

- 组织树、岗位承载约束的业务判断
- 字典或国际化的领域统计逻辑
- 登录日志和操作日志的领域细节

### 3.2 依赖层：`system/org`

`system/org` 负责：

- 部门管理首屏治理链路
- 岗位治理语义
- 组织树、治理任务、岗位承载约束相关查询

它不负责：

- 平台通用分页组件能力
- 低代码工作域头部语义

### 3.3 依赖层：`system/config`

`system/config` 负责：

- 字典管理治理头部语义
- 国际化管理数据读取、治理摘要与缺失统计

它不负责：

- 把低代码页面继续当成配置子页来解释

### 3.4 依赖层：`system/auth`

`system/auth` 负责：

- 登录日志
- 会话管理
- 异常登录与在线会话的联动治理

它不负责：

- 操作日志的领域归属
- 平台级分页与通用头部壳层

### 3.5 依赖层：`system/audit`

`system/audit` 负责：

- 操作日志查询、详情、清理和审计记录
- 高风险操作的安全事件联动

它不负责：

- 直接承载登录主链路或会话建模

### 3.6 非目标

本轮明确不处理：

- 多租户差异化治理
- 重新设计整套左侧导航 IA
- 为所有页面同步做视觉大改版
- 引入新的微服务或独立安全中心产品

## 4. 前 5 项修复设计

### 4.1 右侧功能栏分页增强

#### 4.1.1 问题

当前右侧功能栏分页只覆盖上一页、下一页或页码切换，缺少明确的首页与末页跳转能力，用户在长列表翻页时回跳成本过高。

#### 4.1.2 设计

统一分页组件补齐以下能力：

- `first page`
- `previous page`
- `numbered page jump` 或现有页码跳转
- `next page`
- `last page`

行为约束：

- 在第一页时禁用首页和上一页
- 在最后一页时禁用末页和下一页
- 组件对总页数未知或总量延迟返回的状态保持安全退化

#### 4.1.3 实施原则

- 优先改共享分页组件或右侧功能栏分页封装
- 禁止在各个列表页复制按钮和页码计算逻辑

### 4.2 治理头部补齐子域

#### 4.2.1 问题

岗位管理、字典管理、模块管理的治理头部缺少明确子域；模块生成器缺少治理头部语义，导致页面处于“有治理结构但无归属说明”的半成品状态。

#### 4.2.2 设计

所有治理页头部统一显式表达：

- 主层归属
- 子域归属
- 治理标题
- 治理摘要
- 页面级风险或约束提示

页面归属固定为：

- 岗位管理：`system/org`
- 字典管理：`system/config`
- 模块管理：`platform/lowcode`
- 模块生成器：`platform/lowcode`

#### 4.2.3 根因处理

优先排查：

- 页面模板是否要求显式传 `eyebrow / domain / subdomain`
- 页面元数据注册是否漏配
- 低代码页是否仍被错误沿用配置页模板

修复以“元数据收口”为主，不接受只在 JSX 中手写一行补文案。

### 4.3 部门管理卡顿

#### 4.3.1 根因假设

首屏卡顿高概率来自以下组合：

- 树结构、治理任务、统计、表格、候选负责人数据一起阻塞
- 部门树递归查询或治理统计存在大查询或 N+1
- 页面等待全部数据齐备后才渲染主框架

#### 4.3.2 设计

部门管理首屏拆为分层加载：

- 页面头部与主骨架先渲染
- 树结构与主表格分离加载
- 治理任务、候选负责人、附加摘要按需或延后加载

若后端查询低效，则同步收敛：

- 组织树查询
- 治理统计聚合
- 候选负责人或岗位关联查询

### 4.4 国际化管理卡顿

#### 4.4.1 根因假设

国际化页高概率存在：

- 词条列表与治理摘要首屏混查
- 前端 mount 时做大体量缺失统计和模块扫描
- 关键过滤字段未命中索引

#### 4.4.2 设计

国际化页按职责拆分查询：

- 列表主查询只服务表格和筛选
- 治理摘要单独聚合
- 重计算项尽量移动到后端聚合或改为显式触发

若现有表结构或索引不足，则允许补索引或补辅助字段。

### 4.5 低代码治理语义补齐

#### 4.5.1 问题

模块管理和模块生成器虽然已经形成工作域，但治理语义不完整，导致它们像“工具页”而不是“可治理的平台能力”。

#### 4.5.2 设计

`platform/lowcode` 统一补齐：

- 明确子域标识
- 治理摘要区
- 约束说明
- 生成结果和模块状态的可治理表达

模块生成器不只补静态文案，还应能说明：

- 当前契约校验状态
- 模块注册状态
- 生成链路风险或未完成项

## 5. 安全审计专项设计

### 5.1 总体原则

本轮把安全审计补成 4 条链路：

1. 登录日志可筛、可清、可核查、可升级为安全事件
2. 会话管理可查询、可下线、可联动风险处置
3. 操作日志可查询、可清理、可详情审计、可升级为安全事件
4. 安全事件具备最小完整生命周期

### 5.2 登录日志

#### 5.2.1 查询维度

至少覆盖：

- 账号
- 登录结果
- 时间范围
- IP / 地区
- 设备或终端特征

#### 5.2.2 清理能力

同时支持：

- 按保留策略清理
- 按时间区间人工清理

人工清理入口必须提供时间选择器，而不是只接受固定保留天数。

#### 5.2.3 事件联动

异常登录记录可被提升为待确认安全事件，并保留与原登录记录的关联。

### 5.3 会话管理

#### 5.3.1 查询维度

至少覆盖：

- 用户
- 设备
- IP
- 最近活跃时间
- 会话状态

#### 5.3.2 治理动作

至少覆盖：

- 单会话下线
- 批量下线
- 按用户清理其他会话

#### 5.3.3 风险联动

会话管理需要能够：

- 从异常登录或安全事件回查相关会话
- 对高风险会话执行处置并留下审计

### 5.4 操作日志

#### 5.4.1 查询与详情

至少覆盖：

- 操作人
- 模块
- 动作
- 结果
- 时间
- 来源页面或资源

详情页需要展示足够的请求上下文与结果摘要，不能只有扁平字段列表。

#### 5.4.2 清理能力

同时支持：

- 按保留策略清理
- 按时间区间人工清理

清理动作本身必须写入新的操作审计记录。

#### 5.4.3 事件联动

高风险操作日志可发起或挂接安全事件，不能只停留在备注字段。

### 5.5 安全事件

#### 5.5.1 事件来源

至少支持：

- 登录日志异常
- 会话风险
- 高风险操作日志

#### 5.5.2 生命周期

状态固定为：

- 待确认
- 已确认
- 已误报关闭
- 已处置关闭

#### 5.5.3 核心字段

至少包含：

- 来源类型
- 来源记录 ID
- 风险级别
- 事件摘要
- 确认人、确认时间
- 处置说明
- 关闭人、关闭时间

#### 5.5.4 核心动作

至少包含：

- 确认
- 标记误报
- 关联处置
- 关闭事件

## 6. 数据、接口与权限策略

### 6.1 数据层

允许的结构性变更包括：

- 追加字段
- 追加索引
- 新增安全事件表或关联表
- 收口会话、登录日志、操作日志与安全事件之间的引用关系

原则：

- 优先保持旧数据可读
- 迁移后页面不得依赖“只有新数据才可显示”的脆弱假设

### 6.2 接口层

允许：

- 补充查询参数
- 拆分首屏摘要与主列表接口
- 为安全事件新增独立 DTO 和动作接口

不允许：

- 继续让一个接口同时承担列表、统计、详情和治理动作全部职责

### 6.3 权限层

至少需要重新核对和补齐：

- 登录日志查询 / 删除 / 清理 / 导出
- 会话查询 / 下线 / 批量下线
- 操作日志查询 / 删除 / 清理 / 导出
- 安全事件查询 / 确认 / 关闭 / 误报标记

### 6.4 审计层

以下动作都必须产生审计记录：

- 登录日志人工清理
- 登录日志定向删除
- 会话下线
- 操作日志人工清理
- 操作日志定向删除
- 安全事件确认、误报关闭、处置关闭

## 7. 验证与验收标准

### 7.1 前 5 项验收

至少验证：

1. 右侧功能栏分页支持回到第一页和跳到最后一页
2. 岗位管理、字典管理、模块管理头部显示明确子域
3. 模块生成器显示治理信息，不再是裸工具页
4. 部门管理首屏可分层渲染，不再因全量阻塞出现明显卡顿
5. 国际化管理首屏查询和治理摘要解耦，不再因重计算产生明显卡顿

### 7.2 安全审计验收

至少验证：

1. 登录日志可按时间区间清理
2. 操作日志可按时间区间清理
3. 登录日志异常可发起待确认安全事件
4. 安全事件可确认、误报关闭、处置关闭
5. 会话管理能从风险记录联动回查并执行下线
6. 所有治理动作具备权限控制和审计留痕

### 7.3 视觉与交互验收

本轮所有前端页面改动必须遵守：

- `FRONTEND_UI_SPEC.md`
- 标准系统页壳层节奏
- loading / empty / error / forbidden / submitting 状态完整

若可运行本地页面，应提供实际渲染证据，而不是只做代码层判断。

## 8. 迁移与同步策略

### 8.1 迁移策略

优先采用兼容式迁移：

- 能加字段不先拆表
- 能补索引不先重写查询模型

但若旧设计阻塞安全事件闭环，则允许：

- 新增事件主表
- 引入来源关联字段
- 调整现有清理接口和 DTO 结构

### 8.2 `pantheon-ops` 同步策略

本轮不直接修改 `pantheon-ops`，但最终必须输出同步清单，至少包含：

- 共享分页或页面模板改动
- `system/org` / `system/config` / `system/auth` / `system/audit` 的后端公共能力改动
- i18n、权限点、迁移、事件表结构相关改动
- `ops` 若存在本地 override 需要人工再对齐的页面或接口

---

## 9. 风险与控制

### 风险 1：只修前端表层，不修慢查询和接口过载

控制：

- 卡顿页必须同时排查首屏请求编排和后端查询路径

### 风险 2：安全事件继续做成零散按钮，而不是完整对象

控制：

- 安全事件必须有来源、状态、动作和审计闭环

### 风险 3：低代码页继续漂在 `system/config`

控制：

- 文档和页面头部都明确固定为 `platform/lowcode`

### 风险 4：`pantheon-base` 修完后 `pantheon-ops` 继承遗漏

控制：

- 交付时输出明确同步清单，不只给“请同步 base 改动”这种空话
