# Language Session And Cross-Page Selection Design

English version: [2026-05-16-language-session-and-cross-page-selection-design.en.md](./2026-05-16-language-session-and-cross-page-selection-design.en.md)

---

## 1. 目标

本轮只处理两个已经确认的问题：

1. 统一登录页语言、用户语言偏好、系统默认语言之间的运行时优先级
2. 统一系统域列表页的跨页批量选择行为，保证批量删除、批量启停等动作可在分页场景下稳定执行

本轮不引入租户体系，不新增新的语言配置中心，不新增跨系统域通用删除器。

---

## 2. 设计结论

### 2.1 语言规则结论

运行时语言优先级固定为：

`本次登录选择 > 用户偏好 > 系统默认`

其中：

- `本次登录选择` 属于 `platform` 壳层会话上下文
- `用户偏好` 存储载体仍为 `system_user.preference_json.language`
- `系统默认` 继续来自 `system/config` 的 `i18n.default_language`

### 2.2 批量选择结论

列表页选择集规则固定为：

- 翻页不清空选择集
- 选择集绑定当前查询上下文，而不是当前页
- 改变筛选条件、搜索词、排序条件或点击重置时，清空选择集
- 批量动作基于完整选择集执行，而不是仅基于当前页可见项

## 3. 范围边界

### 3.1 主层：`platform`

本轮主层归属 `platform`，因为两项能力都属于平台壳层的运行时与交互规则：

- 登录前后语言解析与壳层展示语言
- 表格跨页选择的统一交互契约

### 3.2 依赖层：`system/auth`

`system/auth` 负责：

- `/login` 登录主链路
- 登录成功后的会话建立
- 当前登录主体信息返回

`system/auth` 不负责：

- 决定语言最终优先级
- 决定用户长期偏好是否覆盖当前会话

### 3.3 依赖层：`system/config`

`system/config` 继续负责提供：

- `i18n.default_language`

但它只提供系统默认值，不再被解释为“最终必须生效的显示语言”。

### 3.4 依赖层：`system/iam`

`system/iam` 继续负责提供：

- `system_user.preference_json.language`
- 用户、角色、权限、菜单等列表页的批量治理动作

但它只提供长期偏好与列表数据治理能力，不负责当前登录会话语言的最终解释。

### 3.5 非目标

本轮明确不处理：

- 多租户、租户级语言隔离
- 真实 SSO / OIDC 语言协商
- 记住我、多浏览器全局同步语言
- 全站所有列表页一次性重构为统一选择集 Hook

## 4. 语言设计

### 4.1 三层语言来源

#### A. 本次登录选择

定义：

- 用户在 `/login` 页主动切换的语言

语义：

- 表示“当前这次登录流程与当前登录会话要使用的语言”

归属：

- `platform` 壳层运行时状态

它不是：

- 用户长期默认偏好
- 系统设置值

#### B. 用户偏好

定义：

- 当前登录主体在 `system_user.preference_json.language` 中保存的语言偏好

语义：

- 表示“该用户在没有显式选择本次语言时，期望使用的默认语言”

它不是：

- 当前会话必须强制采用的最终语言

#### C. 系统默认

定义：

- `system/config -> i18n.default_language`

语义：

- 表示平台兜底默认语言

它只在：

- 没有本次登录选择
- 没有用户偏好

时生效。

### 4.2 登录前行为

未登录访问 `/login` 时：

1. 平台先读取系统默认语言作为初始显示值
2. 如果登录页提供语言切换，用户切换后立即作用于当前登录页
3. 该选择进入“本次登录会话语言覆盖”上下文

当前阶段不要求登录前去识别“上次是谁登录”，也不允许让上一位用户的会话语言残留决定下一位用户的登录页语言。

### 4.3 登录成功后的行为

登录成功进入系统时：

1. 如果存在“本次登录选择”，则进入系统后继续沿用该语言
2. 该语言不得再被当前用户历史偏好反向覆盖
3. 如果本次未显式选择语言，则尝试读取当前用户偏好
4. 若用户也没有偏好，再退回系统默认语言

也就是说，`GET /auth/me` 或登录返回体中的 `preferences.language` 只能在“本次未显式选语言”时参与决策。

### 4.4 系统内切换语言

系统内切换语言的默认规则固定为：

- 只影响当前会话
- 不自动写入 `preference_json.language`

这样做的目的：

- 避免共享设备、多账号切换时把临时语言切换误写成长期偏好
- 保持“当前会话语言”和“用户默认偏好”语义分离

本轮不要求新增“设为我的默认语言”入口。如果后续需要，可作为二级增强能力再补。

### 4.5 登出行为

登出时必须清理：

- 当前登录会话语言覆盖

登出时不能清理：

- 用户长期偏好字段本身

这样下一位登录人重新进入 `/login` 时，语言解析会重新回到：

- 登录页当前选择
- 当前登录人的偏好
- 系统默认

而不是继续沿用前一位登录人的会话态。

### 4.6 为什么本轮不引入租户

本问题的核心是：

- 当前会话语言归谁控制
- 用户长期偏好何时生效

这属于 `platform + auth + iam + config` 的边界问题，不是租户隔离问题。

租户体系解决的是：

- 数据隔离
- 组织边界
- 独立配置命名空间

它不应作为“每个用户语言不同”的前置条件。

因此本轮结论是：

- **不因语言偏好或列表分页问题提前引入租户**

## 5. 跨页批量选择设计

### 5.1 设计目标

当前系统域存在大量分页列表。用户在做冒烟测试、批量清理、批量启停时，必须能够：

- 在第 1 页选择一批记录
- 翻到第 2 页继续选择
- 最终一次性执行批量动作

因此选择集必须脱离“当前页”而存在。

### 5.2 核心规则

选择集规则固定为：

1. **分页切换不清空**
2. **同一查询上下文内跨页保留**
3. **查询上下文变化时清空**
4. **批量动作基于完整选择集**

### 5.3 查询上下文定义

当前阶段，“查询上下文”至少包含：

- 搜索词
- 筛选条件
- 排序字段
- 排序方向

分页参数 `page / pageSize` 不属于“清空选择集”的条件本身。

解释：

- `page / pageSize` 只是结果窗口变化
- 搜索、筛选、排序变化会改变结果集语义

### 5.4 必须清空的场景

以下动作必须清空选择集：

- 修改筛选条件
- 修改搜索词
- 修改排序字段
- 修改排序方向
- 点击重置
- 批量删除成功
- 批量启用/禁用成功

原因：

这些动作会让原选择集与当前结果集的业务语义脱节，继续保留会增加误删或误操作风险。

### 5.5 不清空的场景

以下动作不得清空选择集：

- 翻页
- 修改每页条数但仍处于同一查询上下文
- 当前页重新加载
- 局部刷新当前页数据

### 5.6 当前页选择框与完整选择集的关系

页面需要同时维护两层状态：

1. **完整选择集**
   - 用于批量操作真实提交
   - 跨页保留

2. **当前页可见选中态**
   - 只负责决定当前页哪些复选框是勾选状态

因此表格的 `selectedRowKeys` 展示值不应等同于“完整选择集原值”，而应取“完整选择集 ∩ 当前页可见数据”。

### 5.7 适用范围

本轮规则优先适用于存在批量治理动作的系统域列表页，例如：

- `system/user`
- `system/role`
- `system/permission`
- `system/post`
- 其他带批量启停、批量删除的治理页

本轮不要求为所有页面统一抽成共享 Hook，但共享规则必须先进入设计与契约。

## 7. 验收标准

### 7.1 语言验收

至少验证以下链路：

1. 未登录打开 `/login`，默认显示系统默认语言
2. 登录页切换语言后，登录成功进入系统，页面保持该语言
3. 若当前用户存在历史语言偏好，也不得覆盖本次登录选择
4. 系统内切换语言后，仅影响当前会话
5. 登出后，下一位登录人不受上一位会话语言污染

### 7.2 跨页选择验收

至少验证以下链路：

1. 在第 1 页勾选若干条记录
2. 翻到第 2 页继续勾选
3. 批量条显示的是完整选择集数量
4. 执行批量删除或批量启停时，提交的是完整选择集
5. 改搜索词、筛选、排序后，选择集被清空

---

## 8. 风险与控制

### 风险 1：语言状态来源继续混写

控制：

- 必须在文档中明确区分“会话覆盖 / 用户偏好 / 系统默认”
- 不允许再用“显式偏好”一个词混指两类状态

### 风险 2：批量选择只修一个页面，规则未沉淀

控制：

- 先把规则写入 `platform` 与 `iam` 文档
- 再按页面逐步实现

### 风险 3：把用户语言问题误扩展为租户问题

控制：

- 文档中明确记录“当前不引入租户”
- 后续只有在数据隔离与配置命名空间出现真实需求时才单独立项
