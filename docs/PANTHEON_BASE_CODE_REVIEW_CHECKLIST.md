# Pantheon Base 企业级后台管理系统 QA 代码审查清单

## 一、审查角色与目标

你是**资深企业级软件QA架构师**，拥有10年以上ToB后台系统审查经验，现在需要对 Pantheon Base 企业级后台底座进行**全维度深度审查**，判断其是否达到生产环境可交付的企业级应用标准。

审查需覆盖：**功能完整性、国际化(i18n)、低代码/动态模块、文件上传存储、前后端契约一致性、安全性、代码质量与命名规范、设计合理性与文档质量、数据库设计、UI/UX设计、质量门禁与CI/CD、版本发布与长期演进、企业级交付能力**十三大维度，严格对照项目设计文档（`DESIGN.md`、`docs/contracts/*`、`docs/designs/*`、`docs/acceptances/*`、`.github/workflows/*`）的契约要求，输出分级问题清单和可落地修复建议。

---

## 二、审查范围

1. **后端代码**：`backend/` 目录所有Go代码，包括 `cmd/`、`internal/`、`pkg/`、`modules/system/*`、`modules/platform/*`
2. **前端代码**：`frontend/src/` 目录所有React/TypeScript代码，包括 `core/`、`components/`、`modules/system/*`、`modules/platform/*`
3. **数据库设计**：`database/migrations/` 所有迁移脚本、表结构、索引设计
4. **设计契约**：根目录 `DESIGN.md`、`docs/` 下所有合同、设计、验收文档
5. **工程配置**：`.github/workflows/*` GitHub Actions工作流、`package.json`/`go.mod`依赖配置、构建脚本、发布脚本
6. **文件结构**：项目目录结构、文件命名、代码注释规范

#### 2. 国际化(i18n)专项检查

对照 `DESIGN.md` 第6章多语言设计规范：

- **核心能力闭环**：
  - [ ] 所有展示文案（菜单/按钮/表单label/错误消息/空状态/确认弹窗/通知提示）全部使用翻译函数，无硬编码自然语言文案
  - [ ] 后端错误码统一返回i18n key，不直接返回中文/英文自然语言错误信息
  - [ ] 菜单/权限点/字典/配置等数据库存储的元数据使用`title_key`/`name_key`字段存储i18n key，不直接存储多语言文案
  - [ ] 语言切换功能正常，切换后全页面实时生效，无需手动刷新页面
  - [ ] 缺失翻译key有降级机制（fallback到默认语言），不直接展示key字符串给用户
  - [ ] 导入导出功能的表头、数据内容、错误提示支持多语言切换
  - [ ] 日期/时间/数字/货币/分页文案等格式化内容符合当前语言环境习惯
- **后端i18n规范**：
  - [ ] 所有返回给前端的提示信息、枚举值、状态标签使用i18n key定义，无硬编码
  - [ ] 审计日志、操作日志存储i18n key而非具体文案，支持按用户语言展示
  - [ ] 提供i18n资源管理接口，支持动态更新语言包无需重启服务
  - [ ] 参数校验错误信息返回i18n key和参数，前端可正确插值翻译
- **前端i18n规范**：
  - [ ] i18n key命名严格遵循域分层规范：`common.*`/`auth.*`/`system.user.*`/`biz.*`等，无混乱命名
  - [ ] 动态渲染内容（枚举值/字典项/状态标签/操作按钮）通过i18n渲染，不硬编码对应文本
  - [ ] 无直接字符串拼接生成文案的情况，支持多语言语序调整（使用插值占位符）
  - [ ] 前端内置默认语言fallback，后端语言包作为增量覆盖，无语言包加载失败导致页面空白
  - [ ] 多语言资源按需加载，无全量加载所有语言包导致的首屏性能问题
  - [ ] 存在缺失翻译key检测机制，构建/运行时可发现未翻译文案

  - [ ] 生成的前端代码可通过TypeScript类型检查

#### 4. 文件上传存储专项检查

对照上传与存储设计规范：

- **核心能力闭环**：
  - [ ] 统一上传入口（POST /api/v1/system/upload）可用，所有业务模块复用该入口

  - [ ] 本地存储文件可通过统一访问地址（GET /api/v1/system/upload/files/*filepath）访问
  - [ ] S3存储模式下正确返回对象访问URL
  - [ ] 个人头像、用户头像上传功能正常可用
  - [ ] 统一封装前端上传方法`uploadSystemFile(file, scope)`，业务模块无需自行实现上传逻辑
- **安全校验**：
  - [ ] 上传文件大小限制生效，超限文件被拒绝
  - [ ] 文件类型白名单校验生效，非允许类型文件被拒绝（默认白名单覆盖jpg/jpeg/png/webp/gif/pdf/doc/docx/xls/xlsx/zip/gz/tgz/tar）

  - [ ] 前端不可自行决定文件大小/类型限制，以后端配置为准
- **审计与可靠性**：
  - [ ] 所有上传操作记录审计日志（上传人、scope、文件名、大小、驱动、结果）
  - [ ] 上传失败场景有明确错误提示，返回i18n key而非硬编码文案
  - [ ] S3配置缺失/初始化失败时错误语义明确，有降级提示
  - [ ] 上传文件URL生成正确，无404/权限问题

- **接口契约一致性**：
  - [ ] 前后端接口路径、请求方法（GET/POST/PUT/DELETE）完全一致，无前端调用不存在的接口
  - [ ] 请求参数（路径参数、query参数、body参数）名称、类型、必填性前后端一致，无参数不匹配
  - [ ] 响应结构（code/msg/data字段、data内数据结构）前后端类型定义完全一致，无字段缺失/类型错误
  - [ ] 错误码、错误提示key前后端一致，前端正确处理所有错误场景
  - [ ] 分页参数（page/pageSize/总数字段）前后端统一，无分页参数不一致问题

- **权限一致性**：
  - [ ] 前端按钮/菜单权限点与后端Casbin权限点完全对应，无前端显示但后端无权限校验的情况

- **枚举值一致性**：
  - [ ] 状态、类型等枚举值前后端定义一致，无枚举值不匹配导致的显示异常/业务错误
  - [ ] i18n key前后端统一，无前端翻译key与后端返回key不匹配问题

#### 6. 安全性检查

- **认证安全**：
  - [ ] 密码是否使用bcrypt/argon2等强哈希算法存储，无明文/弱加密
  - [ ] Token/会话存储安全，无会话固定、会话劫持漏洞
  - [ ] 登出操作是否正确销毁服务端会话，token黑名单机制生效
  - [ ] 敏感操作（修改密码、授权、删除用户、修改系统配置）是否强制二次验证
- **权限安全**：
  - [ ] 不存在水平越权：普通用户无法操作/查看其他用户的数据
  - [ ] 不存在垂直越权：低权限用户无法访问高权限接口/菜单
  - [ ] 前端权限控制仅为展示优化，所有接口后端做独立权限校验
- **输入输出安全**：
  - [ ] 所有用户输入做严格校验（长度、类型、格式、特殊字符）
  - [ ] 无SQL注入：所有数据库操作使用参数化查询，无拼接SQL情况
  - [ ] 无XSS漏洞：用户输入内容正确转义，富文本场景做标签白名单过滤
  - [ ] 文件上传安全：校验文件类型、大小、后缀，无路径遍历漏洞
- **数据安全**：
  - [ ] 敏感配置（数据库密码、密钥、第三方凭证）是否加密存储，无明文
  - [ ] 接口返回敏感字段（密码、密钥、手机号、身份证）是否脱敏或不返回
  - [ ] 安全头配置正确：CSP、HSTS、X-Frame-Options、X-XSS-Protection等
  - [ ] CORS配置安全，不允许所有来源跨域
- **其他安全**：
  - [ ] 无硬编码密钥/密码/凭证在代码中
  - [ ] 生产环境错误信息不返回堆栈、数据库结构等内部敏感信息
  - [ ] 批量操作有数量限制，防止恶意批量请求拖垮服务

  - [ ] 复杂业务逻辑有行内注释说明设计思路，无冗余、过期、错误注释

  - [ ] 低代码、上传存储等公共能力统一抽象到平台层，无业务模块重复实现
- **扩展性设计**：
  - [ ] 动态模块注册机制可用，新业务模块接入无需修改底座核心代码
