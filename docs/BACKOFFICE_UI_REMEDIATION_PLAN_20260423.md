# 后台 UI 专项整改方案

更新时间：2026-04-30

类型：Remediation
归属层：platform
状态：Active

本文用于指导 Pantheon Base 后台管理界面的专项整改。整改重点是**后台管理本身**，覆盖登录页、平台壳层、工作台、系统域管理页、全局状态与反馈，不设计 `business/*` 的业务页面。

## 1. 范围与边界

### 1.1 所属层级

本次整改跨越以下层级：

- `platform`：登录后应用壳层、导航、顶部栏、页签、工作台、主题 token、全局状态页。
- `system/auth`：登录页、认证反馈、当前用户安全入口、登录日志与会话相关页面的体验一致性。
- `system/iam`：用户、角色、权限、菜单等系统管理页面的列表、表格、授权交互一致性。
- `system/org`：部门、岗位等组织类树表与列表页一致性。
- `system/config`：字典、系统设置等配置页一致性。
- `system/audit`：操作日志、登录日志等审计页的信息密度与可读性。

### 1.2 非目标

- 不设计具体 `business/*` 模块页面。
- 不把业务统计硬编码进平台工作台。
- 不改变后端垂直切片与模块边界。
- 不用视觉整改掩盖权限、i18n、状态缺失等结构性问题。

### 1.3 总体判断

当前后台功能闭环已经接近可用，主要短板不是功能数量，而是**从登录页到系统内部的视觉语言不够统一**：部分页面偏营销化、卡片堆叠感明显、渐变与光晕使用过多，容易呈现“AI 生成后台”的模板感。

整改目标是把 Pantheon Base 收敛为：**冷静、可信、工具化、信息密度稳定的企业后台**。

### 1.4 2026-04-30 复检补充结论

本轮复检确认，当时前端不是“只有少数页面未整改”，而是存在明确的**双轨样式状态**：

- 一部分页面已经接入 `PageContainer / PageHeader / FilterPanel / AppModal` 等公共骨架；
- 同时，大量系统页仍在复用旧的 `system-page-side / system-page-summary-card / system-page-note` 右侧栏模板；
- 浮层体系已经有 `AppModal`，但仍有局部页面直接使用 Arco 原生 `Modal` 或 `Drawer`；
- 左侧导航、主内容双栏、右侧辅助栏和浮层容器还没有形成“平台壳层统一治理 + 页面只填内容”的真正闭环。

因此，整改顺序必须调整为：先锁死平台层规则与迁移边界，再做组件收口，最后才是逐页替换旧样式。

补充收口结论：

- 截至 2026-04-30 本轮代码收口完成，旧右栏类名已从页面代码退出；
- `PageSplitLayout / SideRail / AppModal / AppDrawer` 已形成新的平台壳层基线；
- `showAppModalConfirm / showAppModalSuccess / showAppModalError` 已补入平台封装，业务层静态 `Modal.*` 入口已退出；
- 左导航竖版 / 横版的共同视觉策略已收口为同一套导航状态语言，不再保留“侧栏一套、顶栏一套”的分叉风格；
- 后续整改重点不再是“继续消灭旧右栏类名”，而是“统一左导航视觉策略，并阻断新的业务层静态浮层回归”。

## 2. 现状诊断

| 区域 | 当前风险 | 整改方向 |
| :--- | :--- | :--- |
| 登录页 | carousel、指标卡、玻璃拟态、营销 hero 过强 | 改为专业认证控制台，强调身份、安全、环境与反馈 |
| 应用壳层 | 渐变、blur、阴影层级不稳定 | 回到中性 surface、弱边框、清晰导航态 |
| 工作台 | 易变成统计卡片墙，且可能混入业务域 | 平台层只做跨系统摘要与待关注事项，业务卡片必须注册接入 |
| 列表页 | 筛选、表格、操作列、分页节奏不完全一致 | 固化 `PageHeader + FilterPanel + AppTable + PageActions` |
| 配置页 | 左右布局、保存区、敏感配置反馈需要统一 | 使用标准 `ConfigPage` 与吸底提交区 |
| 安全/审计页 | 安全语义与普通 CRUD 视觉差异不足 | 用风险等级、设备态、时间线强化安全上下文 |
| 全局反馈 | loading、empty、error、forbidden 表达不够制度化 | 所有页面显式覆盖五类状态 |

## 3. 目标设计语言

### 3.1 设计关键词

- Calm：冷静，不靠炫技制造高级感。
- Structured：结构优先，页面先服务任务。
- Professional：像企业管理控制台，不像营销落地页。
- Trustworthy：权限、安全、审计场景有明确反馈。
- Low-AI-smell：避免卡片墙、渐变堆砌、假数据装饰。

### 3.2 视觉原则

- **中性色为底**：页面 chrome 使用白色、冷灰、浅边框；Indigo 只用于主操作、选中态和关键反馈。
- **少渐变、少光晕、少玻璃**：登录页和工作台允许极轻品牌图形，但不得成为主视觉。
- **少卡片墙**：卡片只用于真实分组、真实指标或真实操作入口，不用卡片填充空白。
- **密度稳定**：表格、表单、详情页采用一致字号、行高、间距和操作列宽度。
- **控件真实**：没有实现的“记住我、忘记密码、通知、快捷操作”不展示为可点击控件。
- **Arco 优先**：优先用 Arco 组件和主题 token，不扩散无规范手写样式。

## 4. 登录页整改设计

登录页属于 `system/auth`，但视觉 token 与背景策略归属 `platform`。它必须与系统内部共享同一套主题，而不是单独做一个“品牌宣传页”。

### 4.1 页面结构

推荐结构：

```text
AuthShell
  ├── BrandPanel
  │   ├── 产品名 / 环境标识
  │   ├── 简短定位
  │   └── 安全与合规提示
  └── LoginPanel
      ├── 登录标题
      ├── 用户名 / 密码
      ├── 语言 / 主题入口
      ├── 提交按钮
      └── 错误 / loading / locked 状态
```

### 4.2 必须保留

- 清晰的产品标识与当前环境。
- 语言切换与主题切换。
- 登录失败、提交中、网络错误、账号禁用等状态反馈。
- 合理的浏览器自动填充兼容。

### 4.3 必须移除或收敛

- 移除营销式 carousel。
- 移除无真实数据支撑的指标卡。
- 移除玻璃拟态大面积背景。
- 未实现前不展示“忘记密码”入口。
- “记住我”未实现 token 策略前不展示，或明确改成仅记住用户名。

### 4.4 视觉要求

- 背景使用 `color.bg.base` + 极轻品牌图形，不用大面积紫蓝渐变。
- 登录卡片使用 `radius.lg`，阴影不超过 `shadow.md`。
- 主按钮使用纯 Indigo，不用渐变按钮。
- 文案使用 i18n key，禁止硬编码自然语言。

## 5. 应用壳层整改设计

应用壳层属于 `platform`，负责承载动态菜单、顶部能力、页签、内容区和全局反馈，不承载具体业务逻辑。

### 5.1 左侧导航

- 展开宽度稳定在 `220~240px`。
- 选中态使用浅 Indigo 背景 + Indigo 文本。
- 图标统一 `16px` 线性风格。
- 长菜单滚动，不压缩主内容区。
- 折叠态保留 tooltip 与当前态。

补充约束：

- 左侧导航只承担信息架构定位，不承载说明卡、帮助文案、统计卡或风险摘要；
- 深色侧边栏如果继续保留，只能服务于导航识别和品牌一致性，不能做成高装饰、高压迫感的视觉中心；
- 竖版侧栏与横版顶栏模式必须共享同一套菜单层级、选中态和 hover 规则，不能变成两套风格体系。

### 5.2 顶部栏

- 高度固定 `56px`。
- 面包屑、语言、主题、用户菜单、退出登录保持稳定位置。
- icon-only 操作必须有 tooltip。
- 用户菜单包含个人中心、安全中心、退出登录，不把个人入口塞进左侧主导航。

### 5.3 内容区

- 页面 padding 默认 `24px`。
- 主内容宽度与表格滚动策略统一。
- 页面之间不允许出现不同背景色、不同卡片阴影、不同圆角体系。
- 所有全局状态页使用同一套 `StatePage` 视觉。

### 5.4 页签

- 页签只表达已打开页面，不替代面包屑。
- 默认固定工作台页签。
- 关闭、固定、关闭其他等操作要符合企业后台常见行为。
- 页签标题使用 `titleKey`。

## 6. 平台工作台整改设计

工作台属于 `platform` 聚合层。它可以汇总多个系统子域状态，但不能反向侵入 `system/auth`、`system/iam`、`system/org`、`system/config` 的内部职责。

### 6.1 推荐结构

```text
DashboardPage
  ├── StatusStrip       当前平台安全/运行概览
  ├── PrimaryActions    高频管理入口
  ├── AttentionPanel    需要关注的风险、异常、失败登录
  ├── DomainOverview    iam / org / config / audit 摘要
  └── RecentActivity    最近登录 / 最近操作摘要
```

### 6.2 内容规则

- 只展示真实接口数据或明确的空态。
- 平台摘要按系统域聚合，不硬编码业务域。
- 业务模块如果需要展示工作台卡片，必须通过模块注册声明 `dashboard widgets`，并标注归属 `business/*`。
- 图表不是默认必需项；没有趋势价值时用列表、状态条、摘要文本更合适。

### 6.3 去卡片墙规则

- 首页第一屏最多保留 4 个一级关键指标。
- 第二层优先放“需要处理/关注”的内容，而不是继续堆指标。
- 卡片之间必须有信息层级差异，不能全部同尺寸同阴影。
- 快捷入口不超过 6 个，更多入口交给菜单和全局搜索。

## 7. 系统域页面整改设计

### 7.1 `system/iam`

- 用户、角色、权限、菜单统一使用列表页或树表页模板。
- 授权交互要表达“导航权限 / 页面权限 / 操作权限 / 接口权限”的层级，不用单一 `list` 权限兜底。
- 权限码、接口路径等字段可使用等宽字体，但不扩大为整体视觉风格。

### 7.2 `system/org`

- 部门与岗位保持组织管理语义。
- 树表缩进、展开、状态标签、操作列宽度必须统一。
- 不允许删除有子节点或有关联用户的组织节点时只给普通错误提示，必须有可理解的阻断说明。

### 7.3 `system/config`

- 字典与设置采用 `ConfigPage` 或主从布局。
- 左侧分类导航固定宽度，右侧配置面板保持一致标题、说明、保存区。
- 敏感配置使用脱敏、确认、审计说明，不用普通输入框无差别展示。

### 7.4 `system/auth` / `system/audit`

- 安全中心、会话、登录日志、操作日志要强化安全和审计语义。
- 风险状态使用语义色和文本说明，不只靠颜色。
- 时间、设备、IP、用户代理等元信息要排版稳定，可扫描。

### 7.5 右侧辅助栏整改边界

右侧辅助栏属于 `platform` 壳层，不属于任何单一系统域页面的自由发挥区域。

整改时统一遵守：

- 右侧栏只承载“当前上下文”或“真实风险/阻断”；
- 不重复主内容区已有 KPI、摘要卡和说明文案；
- `system/iam` 默认轻量摘要，`system/config`、`system/auth`、`system/audit` 才允许使用风险型右栏；
- 当页面不满足右栏职责时，直接退回单列页面，而不是为了版式对称强行保留一栏；
- 右栏卡片数和说明文案长度受限，不再接受“第二主内容列”式堆叠。

## 8. 组件与样式整改清单

### 8.1 P0 必改

- 登录页重做为专业认证控制台。
- 应用壳层去除过度渐变、glass blur 和不稳定阴影。
- 工作台去卡片墙，按平台层聚合结构重排。
- 所有装饰性但未实现的控件必须移除、禁用并说明，或补齐真实行为。
- 将登录页、壳层、系统页共用同一套 theme token。
- 停止新增旧的 `system-page-side / system-page-summary-card / system-page-note` 页面模板。
- 停止在 `platform`、`auth`、`system/*` 页面中继续直接新增原生 `Modal` / `Drawer`。

### 8.2 P1 强化

- 统一系统列表页的筛选、表格、分页、操作列、批量操作。
- 统一 Modal、Drawer、Message、Notification 的圆角、阴影、footer。
- 统一安全中心、会话、登录日志、操作日志的信息密度。
- 补齐全局 403、404、500、网络错误、搜索空态视觉。

### 8.2.1 当前混合态遗留清单

以下模式视为“历史遗留待收口”的上一阶段盘点结果；截至 2026-04-30 的当前代码状态，右栏和主要浮层链路已完成退出旧模板，不再继续按此清单判定页面为 Legacy。

- 当前仍需持续治理的例外点：
  - `Modal.confirm / success / error` 仍存在于 `AppModal` 平台封装内部，这是允许的统一入口，不再允许业务层直接调用
  - 左导航与横版顶栏虽然已共用一套状态语言，但仍需在每轮壳层验收中同时检查两种布局，避免只改一种
  - `auth/login-log`
- 直接使用原生 `Modal` 的页面：
  - `system/config -> dynamicmodule`
  - `generator/*` 局部能力
  - `components/feedback` 个别二次验证浮层
- 直接使用原生 `Drawer` 的页面：
  - `generator/components/CodePreview`

这些实现允许短期并存，但必须在迁移计划里被明确标注，不能默认视为“整改已完成”。

### 8.3 P2 演进

- 支持用户级主题、导航模式、表格密度偏好。
- 建立 dashboard widget 注册机制。
- 建立截图基线与视觉回归清单。

## 9. 页面级验收标准

每个后台页面整改后必须满足：

- 页面类型明确：`AuthPage / DashboardPage / ListPage / TreePage / ConfigPage / DetailPage / StatePage`。
- 文案全部走 i18n key。
- 显式处理 `loading / empty / error / forbidden / submitting`。
- 主操作、危险操作、批量操作有明确权限点。
- 组件优先使用 Arco Design 与公共 pattern 组件。
- 不出现无真实行为的伪控件。
- 不出现大面积渐变、玻璃拟态、营销 hero、三列 feature card 等 AI 味套路。
- 登录页到系统内部视觉 token 一致。

额外平台层验收：

- 页面若存在右侧栏，必须能回答“它承载的是 Context 还是 Alert”，否则判定为设计不通过；
- 页面若使用弹窗或抽屉，必须能回答“为什么不是独立页面”以及“为什么不是另一种浮层形态”；
- 页面若仍沿用旧右栏类名或原生浮层，必须被明确标记为历史遗留，而不是默认接受。

## 10. 推荐实施顺序

1. 先统一 token 与壳层基础样式，避免后续页面继续扩散旧风格。
2. 重做登录页，作为“进入系统前”的第一张专业名片。
3. 重排工作台，先解决平台层聚合边界和卡片墙问题。
4. 先抽象壳层级右侧辅助栏和浮层规则，再批量收敛系统域列表页与配置页的页面骨架。
5. 补全安全、审计、异常态页面的专业表达。
6. 做一次“旧类名 / 原生浮层 / 双轨样式”全局扫描，确认没有新增扩散。
7. 用截图和冒烟用例固定整改结果，防止回退。

## 11. 验证方式

- 视觉走查：登录页、工作台、用户、角色、菜单、权限、部门、岗位、字典、设置、安全中心、日志页逐页截图。
- 状态走查：每类页面至少验证 loading、empty、error、forbidden、submitting。
- 权限走查：普通管理员、只读角色、无权限用户三类账号检查导航、页面、按钮、接口反馈。
- 多语言走查：中英文切换后标题、按钮、表格列、空态、错误提示不溢出。
- 构建验证：前端执行 `npm run build`，后端无需因 UI 文档整改执行数据库迁移。

补充扫描项：

- `rg "system-page-side|system-page-summary-card|system-page-note"`：确认旧右栏模式是否还在扩散；
- `rg "<Modal|<Drawer"`：确认是否仍有页面绕过统一浮层模式；
- 左导航需同时验证竖版侧栏和横版顶栏两种布局模式，不接受只验一种；
- 右侧栏页面需验证桌面与中屏退化，确认不会形成“双列挤压”。

补充验收纪律：

- 凡是改动 `platform` 壳层导航相关代码，默认必须提交“双模式验收记录”；
- 双模式验收记录至少覆盖：竖版展开、竖版折叠、横版主导航、横版弹出子菜单；
- 没有双模式验收记录的壳层改动，只能视为“待验收”，不能直接视为整改完成；
- 如果某一布局模式暂时无法通过，必须在矩阵文档中以 `Pending` 明确挂账，不接受口头说明。
- 双模式验收记录统一使用 `docs/PLATFORM_SHELL_DUAL_MODE_ACCEPTANCE_TEMPLATE.md`。
