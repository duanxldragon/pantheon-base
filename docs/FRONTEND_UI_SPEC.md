# 前端 UI 详细设计规范

更新时间：2026-04-17

本文是 Pantheon Base 的前端 UI 细则文档，用来补足 `docs/FRONTEND.md` 中尚未展开的视觉、交互、响应式、状态、导航和页面模板规范。

目标不是写一份“好看的设计描述”，而是沉淀一份**后续可以直接排期、直接指导页面开发、直接约束 AI 生成代码**的标准。

## 1. 设计目标

- **标准企业后台感**：克制、稳定、清晰、可信。
- **高复用**：所有 CRUD 页、详情页、配置页使用统一页面骨架。
- **多语言友好**：长文本、双语长度差异、按钮文案膨胀都能承受。
- **权限友好**：页面必须能表达无权限、部分权限、只读等状态。
- **响应式可控**：优先桌面，但要有明确的中小屏退化方案。

## 2. 设计基因

### 2.1 关键词

- Indigo
- Enterprise
- Clean
- Structured
- Calm
- Figma-inspired
- Low-AI-smell

### 2.1.1 多主题策略（2026-04-20 补充）

Pantheon Base 的主题能力归属 **platform 层**，不是某个单独系统域页面私有的视觉配置。

当前统一约束：

- 登录页、工作台、系统管理页、对话框、Drawer、Message、Notification 共用同一套主题 token。
- 主题切换只改 token，不复制一套新的页面结构。
- 默认保留企业后台的克制感，允许在品牌主色、背景气质、侧边栏氛围上做差异化。

首批主题建议固定为：

- `indigo`：默认企业后台主题
- `emerald`：偏协作/运营氛围
- `violet`：偏平台化/智能化氛围
- `slate`：偏低饱和稳态办公氛围

### 2.2 视觉原则

- **少装饰、重结构**
- **弱分割、强节奏**
- **可扫描优先于花哨**
- **操作安全感优先于视觉冲击**
- **界面 chrome 克制，业务内容出彩**
- **层级靠字号、字重、密度、留白建立，不靠大面积渐变**

### 2.3 参考来源与取舍

本项目参考 `awesome-design-md` 的做法：把设计系统写成 AI 可读的 Markdown，让 AI 在明确约束内生成 UI，而不是每次自由发挥。

参考方向：

- `awesome-design-md` 的文档结构：视觉氛围、颜色角色、字体层级、组件状态、布局原则、禁忌清单、响应式行为。
- Figma 风格条目：黑白界面骨架、轻量字体层级、pill/circle 几何、彩色内容作为点缀、虚线 focus 反馈。
- 企业级后台产品：信息密度、稳定性、权限态、表格可读性、审计可追踪性。

取舍原则：

- **借鉴 Figma 的克制和精确，不复制 Figma 的营销页风格。**
- **保留企业后台的稳定感，不做过度 playful 的多彩界面。**
- **彩色只作为状态、强调、插画或工作台图形点缀，不侵入基础操作 chrome。**
- **所有设计规则必须能落到 Arco Design + token + 组件封装，不停留在描述层。**

## 3. 设计 Token

### 3.1 颜色

| Token | 值 | 用途 |
| :--- | :--- | :--- |
| `color.primary` | `#165DFF` | 品牌主色、主要 CTA |
| `color.bg.base` | `#F7F8FA` | 应用底色 |
| `color.bg.surface` | `#FFFFFF` | 卡片、表格、筛选区 |
| `color.border.default` | `#E5E6EB` | 默认边框 |
| `color.text.primary` | `#1D2129` | 主文本 |
| `color.text.secondary` | `#4E5969` | 次级文本 |
| `color.text.tertiary` | `#86909C` | 辅助说明 |
| `color.success` | `#00B42A` | 成功 |
| `color.warning` | `#FF7D00` | 警告 |
| `color.danger` | `#F53F3F` | 危险 |

### 3.1.1 Figma-inspired 颜色策略

Figma 的一个关键启发是：**界面本身尽量黑白中性，颜色留给内容和关键反馈。**

Pantheon Base 不采用纯黑白极端方案，而采用企业后台更稳的版本：

- 基础界面使用冷灰、白色、浅边框和少量 Indigo。
- 主按钮和选中态使用 Indigo，不做渐变按钮。
- 图表、统计卡、状态标签允许使用彩色，但必须有明确语义。
- 页面背景不使用花哨渐变；登录页、工作台可以有极轻微品牌图形。
- 禁止默认使用紫蓝大渐变、彩色圆形 icon 卡片、玻璃拟态背景。

### 3.2 圆角

| Token | 值 | 场景 |
| :--- | :--- | :--- |
| `radius.xs` | `8px` | Input / Tag |
| `radius.sm` | `12px` | Button / Input / 小卡片 |
| `radius.md` | `16px` | Card / Modal / Drawer |
| `radius.lg` | `22px` | 登录卡片 / 工作台重点卡片 |

实现层语义映射：

- `radius.control`：`8px`，用于 Input / Select / Picker / Textarea，避免表单控件过度胶囊化。
- `radius.action`：`12px`，用于 Button，保留明确可点击感。
- `radius.overlay`：`16px`，用于 Modal / Drawer 外层壳子，保持浮层克制、稳定。

### 3.3 阴影

| Token | 值 | 场景 |
| :--- | :--- | :--- |
| `shadow.sm` | `0 1px 2px rgba(0,0,0,0.05)` | 普通卡片 |
| `shadow.md` | `0 6px 16px rgba(0,0,0,0.08)` | 下拉 / 抽屉 |
| `shadow.lg` | `0 12px 32px rgba(22,93,255,0.08)` | 重点悬浮层 |

### 3.4 间距

统一采用 4px 基线：

- `4 / 8 / 12 / 16 / 24 / 32 / 40 / 48`

推荐：

- 页面 padding：`24`
- 筛选区和表格区间距：`16`
- 模块区块间距：`24`
- 重要卡片组间距：`24` 或 `32`

### 3.5 字体层级

| Token | 值 |
| :--- | :--- |
| `font.size.xs` | `12px` |
| `font.size.sm` | `14px` |
| `font.size.md` | `16px` |
| `font.size.lg` | `20px` |
| `font.size.xl` | `24px` |

字重：

- 页面标题：`600`
- 区块标题：`600`
- 表格正文：`400`
- 强调数字：`600`

### 3.5.1 字体风格策略

Figma 风格的核心不是“花字体”，而是细腻字重和紧凑排版。Pantheon Base 采用更适合企业后台的版本：

- 字体优先使用系统字体或企业可控字体，不强依赖外部字体服务。
- 标题使用 `600`，避免过粗。
- 表格和表单正文保持 `14px / 400`，保证信息密度。
- 工作台大数字可用 `24px~32px / 600`。
- 标签、权限码、接口路径可使用等宽字体，但不大面积使用 monospace。
- 允许轻微负字距只用于大标题，不用于表格和长文本。

## 4. 总体布局规范

### 4.1 应用壳层

后台基础布局默认采用左侧导航，但平台壳层必须支持用户在“竖版侧边导航 / 横版顶部导航”之间切换：

- 左侧侧边导航（默认）
- 顶部横版导航（用户偏好）
- 顶部全局栏
- 主内容区
- 底部版权区（可弱化）

导航布局模式归属 **platform 层**，只改变动态菜单的呈现位置，不改变菜单数据、路由注册、权限判断和业务模块装配契约。用户切换结果应作为前端壳层偏好持久化，短期可使用 `localStorage`，后续可接入平台设置或用户偏好接口。

横版顶部导航不是“裸菜单平铺”，必须具备独立的 surface 底色、弱边框、滚动容器、hover 态和选中态；子菜单、外链菜单与普通菜单应共用同一套视觉规则，避免切换到横版后丢失底色和当前菜单识别。

### 4.2 侧边栏

- 展开宽度：`220~240px`
- 折叠宽度：`56~64px`
- 背景：`#FFFFFF`
- 右侧分割线：`1px solid var(--color.border.default)`

必须支持：

- 折叠
- 高亮当前菜单
- 二级菜单展开
- 长菜单滚动
- 图标 + 文案

### 4.2.1 侧边栏视觉细节

- 菜单项高度建议 `40px`。
- 菜单项圆角建议 `6px`，不使用夸张大圆角。
- 选中态使用浅 Indigo 背景 + Indigo 文本，不使用高饱和色块。
- 图标使用线性图标，大小 `16px`。
- 二级菜单缩进稳定，不因文本长度跳动。

### 4.3 顶部栏

至少包含：

- 面包屑
- 主题切换
- 语言切换
- 用户信息
- 个人中心入口
- 退出登录

可扩展：

- 全局搜索
- 消息通知
- 快捷操作入口

### 4.3.1 顶部栏视觉细节

- 顶部栏高度建议 `56px`。
- 面包屑使用 `13px~14px`，弱化非当前项。
- 顶部按钮尽量 icon-only + tooltip，避免挤压主内容。
- 用户区不做复杂卡片，只保留头像/用户名/下拉。
- 主题切换与语言切换并列放置，属于平台层全局能力，不要散落到具体业务页面中各自实现。

## 5. 导航与信息架构规范

### 5.1 一级导航建议

- 工作台
- 访问控制
  - 用户
  - 角色
  - 权限
  - 菜单
- 组织架构
  - 部门
  - 岗位
- 平台配置
  - 字典
  - 系统设置
  - 多语言资源
- 安全审计
  - 登录日志
  - 会话管理
  - 操作日志

### 5.2 面包屑规则

- 首页固定为第一项
- 后续应根据菜单树生成完整父链
- 个人中心、安全中心这类“非左侧主导航页”允许单独注册 breadcrumb 元数据

补充约束：

- 左侧主导航承载管理员视角的全局治理页。
- 当前用户的个人中心、安全中心通过右上角用户菜单进入，不占左侧一级导航。

### 5.3 页签策略

平台壳层应区分“面包屑”和“已打开页签”：

- 面包屑只表达当前页面在信息架构中的父链，不承载多页面状态。
- 已打开页签承载最近打开的菜单/页面，用于在多个系统页面之间快速切换。
- 仪表盘页签默认保留，其他页面可关闭；关闭当前页签后回退到最近一个已打开页签。
- 页签应支持右键快捷菜单，至少包含“关闭当前 / 关闭其他 / 关闭右侧”三类企业后台常见批量操作。
- 页签应支持“双击关闭、固定页签、关闭全部”；其中固定页签默认靠左展示，且不参与“关闭其他 / 关闭右侧 / 关闭全部”。
- 页签应支持拖拽排序与中键关闭；拖拽排序默认只在同一分组内生效，避免破坏“固定页签在左、普通页签在右”的平台壳秩序。
- 页签标题优先使用路由或菜单的 `titleKey`，避免硬编码自然语言。

设计上继续预留：

- 当前页签
- 最近访问
- 收藏页

## 6. 页面骨架规范

### 6.1 标准列表页

必须统一为：

1. 页面标题区
2. 操作区
3. 筛选区
4. 表格区
5. 分页区

推荐结构：

```text
PageHeader
ActionBar
FilterCard
TableCard
Pagination
```

### 6.1.1 列表页视觉细节

- 页面标题和主操作在同一行。
- 筛选区使用白色 surface，不使用彩色背景。
- 表格、筛选区、分页、空态卡片统一继承主题圆角与阴影，不在单页手写另一套视觉语言。

### 6.2 标准详情页

用于用户详情、策略详情、系统设置详情：

1. 标题区
2. 基本信息卡
3. 分区信息块
4. 底部操作区

### 6.3 标准配置页

用于系统设置、字典配置：

- 左侧分类导航
- 右侧配置面板
- 吸底保存区

### 6.4 标准工作台页

仪表盘至少包含：

- 统计卡片
- 快捷入口
- 待处理事项
- 最近操作
- 告警/异常摘要

### 6.5 Figma-inspired 组件形态

可借鉴：

- Pill 形态用于筛选标签、状态分段控件、二级 tab。
- Circle 形态用于 icon-only 操作，如语言切换、刷新、更多。
- 虚线 focus outline 可作为可访问性特色，但必须统一。
- 工作台图形区可使用轻量多彩点缀，主操作区保持中性。

不要照搬：

- 大面积多色渐变 hero。
- 营销页式超大标题。
- 过度轻薄字重导致后台表格不可读。
- 把黑白极简用于所有业务页，导致状态区分不足。

## 7. 表单规范

### 7.1 栅格

- 桌面端默认 24 栅格
- 常见字段跨度：`6 / 8 / 12 / 24`
- 必须预留 `xs / sm / md / lg` 响应式配置

### 7.2 标签与帮助文案

- 表单标签左对齐，长度可控
- 帮助文案放在字段下方，不写在 placeholder 里
- placeholder 只做输入示例，不承载规则说明

### 7.3 校验

必须区分：

- 必填
- 格式错误
- 业务规则错误
- 提交失败

错误文案统一走 i18n key。

### 7.4 危险字段

以下字段需要更强交互：

- 密码
- 状态停用
- 删除
- 权限变更
- 系统级配置

要求：

- 二次确认
- 更明确的 warning/danger 视觉提示

## 8. 表格规范

### 8.1 列类型

常见列：

- 文本
- 状态
- 时间
- 标签
- 关联对象
- 操作列

### 8.2 操作列

默认顺序：

- 查看
- 编辑
- 删除

如果有更多操作：

- 超过 3 个操作后折叠到 `更多`

### 8.3 表格空态

必须区分：

- 初始空态
- 搜索无结果
- 无权限空态
- 加载失败

### 8.4 批量操作

以下页面建议支持批量操作：

- 用户
- 角色
- 权限策略
- 字典项

## 9. 状态设计

所有页面必须定义以下状态：

- `loading`
- `success`
- `empty`
- `error`
- `forbidden`
- `submitting`

### 9.1 `loading`

- 页面级使用 skeleton 或 spin
- 表格级使用表格 loading
- 不允许整页闪烁

### 9.2 `empty`

要区分：

- 首次使用空态
- 筛选后无结果

### 9.3 `error`

至少覆盖：

- 网络错误
- 服务器错误
- 请求超时
- 数据加载失败

### 9.4 `forbidden`

必须有统一 403 表达：

- 页面级无权限
- 操作级无权限

### 9.5 Focus 与可访问性

参考 Figma 的编辑器感反馈，Pantheon Base 统一要求：

- 所有可交互元素必须有清晰 focus 态。
- 键盘 focus 可使用 `2px dashed` 或 `2px solid`，但必须全局一致。
- 不允许只靠颜色表达状态。
- 表单错误必须同时有颜色、文本和布局反馈。
- 表格操作按钮必须有可读 tooltip 或文本。

## 10. 权限交互规范

### 10.1 三层权限态

- **导航权限**：能否看到菜单
- **页面权限**：能否进入页面
- **操作权限**：能否执行按钮/接口动作

### 10.2 页面无权限

路由级无权限时，不应只靠接口 403 后报错。

正确行为：

- 路由进入前校验
- 或进入后渲染统一 403 页面

### 10.3 按钮无权限

- 默认直接隐藏
- 高风险或教育场景可显示 disabled + tooltip

## 11. 多语言设计规范

### 11.1 文案规则

所有用户可见文案必须通过 i18n 输出，包括：

- 页面标题
- 表格列名
- 按钮
- 提示文案
- modal 文案
- 错误文案
- 空态文案

### 11.2 Key 命名

建议结构：

- `common.*`
- `auth.*`
- `system.user.*`
- `system.role.*`
- `system.permission.*`
- `biz.order.*`

### 11.3 长文本兼容

必须考虑：

- 英文比中文更长
- 德文可能更长
- 按钮区不能因为翻译长度直接撑爆布局

## 12. 响应式规范

### 12.1 断点

建议：

- `xs < 576`
- `sm >= 576`
- `md >= 768`
- `lg >= 992`
- `xl >= 1200`

### 12.2 中小屏退化策略

- 筛选区折叠
- 表单单列化
- 表格允许横向滚动
- 侧边栏抽屉化

### 12.3 当前阶段策略

Pantheon Base 当前优先桌面端，但代码不能写死成“只有桌面端能用”。

## 13. 动效规范

- 页面切换：轻量淡入，不做夸张转场
- Drawer / Modal：统一 easing 和 duration
- Hover：轻微即可
- Danger 操作：强调反馈而非娱乐性

## 14. 全局体验统一规范（2026-04-23 补充）

本节承接 `docs/BACKOFFICE_UI_REMEDIATION_PLAN_20260423.md`，用于约束登录页到系统内部的一致体验。

### 14.1 Modal / Drawer

- 统一使用 `radius.md`；
- Header / Footer 分割线使用标准边框色；
- 表单型弹窗与详情型抽屉沿用同一套 panel 背景与 shadow；
- 危险操作只通过按钮语义和提示色强调，不额外制造刺眼背景。

### 14.2 Message / Notification

- 统一使用轻阴影 + 中圆角；
- 成功、警告、错误保留语义色，但背景不要大面积纯色铺满；
- 文案仍遵循 i18n key 约束，禁止写死自然语言。

### 14.3 登录页与应用壳一致性

- 登录页品牌色、背景层次、圆角、阴影和主题切换必须与应用壳共享同一批 token；
- 登录页是认证控制台，不是营销页；不得默认使用 carousel、虚假指标卡、大面积渐变 hero 或玻璃拟态；
- 未实现真实行为的控件不展示为可点击能力，例如“忘记密码”“记住我”“通知中心”等；
- 主题切换后，登录页与系统管理页要同时变化，避免“登录页一套、后台一套”的割裂感；
- 登录错误、提交中、网络异常、账号禁用、锁定等状态必须有统一反馈。

### 14.4 应用壳一致性

- 侧边栏、顶部栏、页签、内容区背景必须使用中性 surface 与标准边框；
- 筛选按钮右对齐，顺序为：重置、查询；
- 表格和筛选区间距 `16px`；
- 操作列固定在右侧，宽度稳定；
- 表格密度默认 `middle`，系统设置允许未来调整密度。

### 14.5 平台工作台一致性

- 工作台属于 `platform` 聚合层，不属于任何单一系统子域；
- 首页不做卡片墙，第一屏最多保留 4 个一级关键指标；
- 第二层优先展示待关注事项、风险摘要、快捷入口和最近活动；
- 不硬编码业务域卡片；业务模块如需接入，必须通过注册机制声明。

## 16. 去 AI 味设计守则

后续 AI 生成 UI 时，必须避开下面这些“AI 味”：

### 16.1 禁止的常见套路

- 紫蓝渐变背景作为默认高级感。
- 3 列 feature card + 彩色圆 icon 的通用模板。
- 所有卡片同样圆角、同样阴影、同样间距。
- 页面内容全部居中。
- 超大 hero 标题塞进后台系统。
- 渐变按钮作为主 CTA。
- 没有真实信息架构，只靠卡片堆叠撑页面。
- 表格页做成营销页风格。
- 登录页做成宣传落地页，而不是认证控制台。
- 用不可点击或未实现的控件制造“功能丰富”的错觉。

### 16.2 应该坚持的企业后台质感

- 页面先解决任务，再考虑装饰。
- 表格列、筛选项、操作按钮有稳定对齐。
- 空态、错误态、无权限态有明确文案。
- 图标少而准，不把 icon 当装饰噪音。
- 颜色少而有语义。
- 动效短而明确，不拖慢操作。
- 每个页面看起来像同一个系统，而不是 AI 拼出来的不同模板。

### 16.3 Figma 风格的正确借鉴方式

- 借鉴它的“工具感”：干净、精确、轻量、组件化。
- 借鉴它的“内容优先”：界面骨架克制，业务内容成为视觉主角。
- 借鉴它的“几何语言”：pill、circle、精准 focus、干净 tab。
- 不借鉴营销页大色块，不让后台变成品牌官网。

## 17. AI 生成代码约束

当 AI 生成前端页面时，必须遵守：

- 先判断页面类型：列表 / 详情 / 配置 / 工作台
- 直接套用对应页面骨架
- 所有文案使用 `t()`
- 所有状态显式处理
- 不新增无规范内联样式
- 权限点、路由、菜单、i18n key 同步补齐
