# 低代码生成链路交付就绪度评估（2026-05-19）

English version: [LOWCODE_DELIVERY_READINESS_20260519.en.md](./LOWCODE_DELIVERY_READINESS_20260519.en.md)

本文评估 Pantheon Base 当前“低代码”相关能力的真实交付状态。

本报告不把 `generator`、`dynamicmodule` 再归回 `system/config`，而按当前代码和主文档口径评估：

- `platform.lowcode`：低代码工作域入口
- `system/generator`：受控代码生成能力
- `system/dynamicmodule`：模块接入治理能力

---

## 1. 评估问题

本轮只回答四个问题：

1. 当前低代码链路已经完成了什么
2. 还缺哪些关键能力
3. 现阶段的主要缺陷是什么
4. 以当前状态看，是否已经可交付

---

## 2. 结论摘要

一句话判断：

> 当前能力已经可以作为 **内部受控脚手架生成器 / 业务模块接入加速器** 交付试点，但还不能作为 **成熟运行时低代码平台产品** 对外宣称可交付。

按交付口径拆开判断：

- 作为内部 Beta：
  - **可交付**
  - 前提是明确产品名义是“受控生成链路”或“研发加速器”
- 作为面向业务实施、运营或非研发角色的低代码平台：
  - **不可交付**
  - 当前仍缺运行时编排、完整前端页面生成、多表业务闭环和更强的服务端收口

---

## 3. 已完成能力

### 3.1 入口与治理边界

当前代码已经把低代码入口从旧的 `system/config` 叙事中抬出来，形成独立工作域：

- `/system/generator` 挂到 `platform.lowcode`
- `/system/modules` 挂到 `platform.lowcode`
- 生成能力和模块治理能力仍保持分离

这意味着当前不是“系统设置附属页”，而是一个独立工作域下的两段链路：

- `generator` 负责生成
- `dynamicmodule` 负责接入治理

### 3.2 生成主链路

当前已经打通以下闭环：

- 模块 schema 编辑
- 受管数据源元数据读取
- 菜单、权限、i18n、关系契约预览
- 代码文件预览
- ZIP 下载
- 生成源码到工作区
- 重写 generated registry
- 写入 `system_module_registration`
- 生成后进入“待激活”状态
- 生成器可显式标记 `metadata.autoRecycle`，用于临时 QA / smoke 模块生命周期治理
- 模块治理页可展示“临时模块”统计、生命周期标签与自动回收提示
- many-to-many 关系已补运行时模板，主从表详情页已补子表新增 / 编辑 / 提交流程

### 3.3 治理与诊断

当前已具备：

- 待激活 / 已接入 / 已卸载 / 失败 状态模型
- 模块注册、卸载、删除记录、清理、修复
- 注册表一致性诊断
- 待激活模块的激活检查与状态回写
- 生成结果摘要与校验结果回写

这说明它已经不是纯前端 Demo，而是具备一定治理能力的工程链路。

---

## 4. 关键缺口

### 4.1 不是运行时低代码

这是当前最核心的事实。

生成完成后，模块不会立即运行时生效，而是先进入“待激活”，仍需要：

- 重启后端
- 重建前端

这决定了它本质上更接近：

- 代码生成器
- 模块装配治理器

而不是：

- 运行时 DSL 平台
- 在线页面编排平台
- 面向业务人员的低代码产品

### 4.2 前端生成物已补到基础 CRUD，但仍停留在骨架层

当前前端生成物已经覆盖：

- `index.ts`
- `api.ts`
- `*List.tsx`
- `*Form.tsx`
- `*Detail.tsx`

目前仍缺：

- 真正可操作的关系表管理页与主从表组合视图
- 主详情页对子表区域的运行时编排
- 可复用表单 schema 渲染层

也就是说，当前已经不是“只有列表页”的起步版；它已经能生成 `List / Form / Detail`，并在页面中带出依赖、关系、主表上下文、关联入口占位，以及基于 `lookupApi` 的 relation options 加载模板。但这些仍然是脚手架层引导，不是可编排的运行时页面体系。

### 4.3 多表业务只有契约，没有真正业务闭环

当前已经补了不少多表元数据：

- `businessContext`
- `tableRole`
- `primaryTable`
- `relationFromField`
- `relationToField`
- `dependencies`
- `relations`

但这些目前仍主要用于：

- 生成前治理
- 契约说明
- 风险阻断

还没有真正进入：

- 主从表 UI 编排
- 多表事务生成
- 详情页子表区块
- lookup / many-to-many 运行时交互闭环

本轮补齐后，生成页已经会显式暴露：

- 关系契约摘要
- 主表上下文标签
- 关联模块入口占位动作
- lookup 选项加载契约与前端模板

但这些入口目前仍然只是“告诉开发者下一步应该接到哪里”，不是系统自动完成的业务交互。

### 4.4 服务端生成权已经收口，但运行时激活仍未收口

当前后端已经收口了两件关键事：

- 预览文件由服务端统一基于 schema 生成
- 生成注册接口不再依赖客户端提交 `files[]`
- 源码包下载也走服务端统一生成链路

这意味着“生成结果以服务端模板为准”这条原则已经基本成立。

当前真正还没收口的是激活阶段：

- 生成后仍需后端重启
- 生成后仍需前端构建
- 缺少更强的自动化激活或集中式执行器
- 真实 E2E 仍依赖“生成源码后重新启动后端”的环境编排，运行时激活体验还不算产品级

所以这里的主要问题已经从“生成信任边界”转成“激活闭环还不够产品化”。

---

## 5. 现阶段主要缺陷

### 5.1 产品定位容易被误读

如果直接把当前能力命名为“低代码平台”，外部理解会默认期待：

- 运行时建模
- 页面可视化
- 流程编排
- 在线发布
- 非研发角色可用

而当前实现并不支持这些核心预期。

因此当前最大的产品缺陷之一不是代码 bug，而是命名风险。

### 5.2 `ModuleWizard` 单页复杂度偏高

生成器工作台已经可用，但单页承载信息过多：

- schema
- 字段建模
- 菜单与权限
- i18n 预览
- 契约摘要
- 数据源管理
- 文件预览

这使它更像一页高密度工程工具，而不是成熟、低门槛的产品化工作台。

### 5.3 生成与激活之间仍然依赖人工运维动作

“待激活”是合理的治理设计，但从交付视角看，这里仍有明显断点。当前已经补了“激活检查”这一步，系统可以在模块治理页或生成结果页重新审计，并把满足条件的模块回写为“已接入”；但真正的激活动作本身仍依赖外部执行：

- 生成后需要人工重启 / 重建
- 缺少更产品化的激活引导
- 缺少更强的自动化激活或集中式执行体验

这会直接影响交付时的使用顺滑度。

### 5.4 英文文档 companion 仍有残余漂移

本轮主文档口径已经完成修正，但英文 companion 和部分历史评估文档还没有同步这次归属调整。

这不是运行时缺陷，但会影响长期文档一致性。

---

## 6. 交付判断

### 6.1 可以交付的口径

以下口径下，我认为当前已经可以交付：

- 内部研发团队使用
- 受控业务模块脚手架生成
- 业务模块接入试点
- 开发环境 / 内部治理环境

前提说明必须写清楚：

- 仅支持 `business/*` 一键生成并注册
- 生成后仍需重启后端与重建前端
- 默认定位为工程加速器，不是运行时低代码平台

### 6.2 不可交付的口径

以下口径下，我认为当前还不能交付：

- 对外宣称成熟低代码平台
- 面向实施顾问或运营角色的可视化建模平台
- 面向非研发用户的在线模块搭建平台
- 支持复杂多表业务编排的产品级低代码系统

---

## 7. 补全优先级

### P0

必须优先补的：

- 服务端模板收口：尽量从“前端生成文件、后端代写”转向“前端传 schema、后端统一生成”
- 前端补齐标准 `Create / Edit / Detail` 生成模板
- 把生成后激活链路做成更清晰的操作闭环和诊断引导

### P1

第二层应补：

- 多表业务的主从表 / 关系表页面模式
- many-to-many 的前端交互模板与统一业务 options 约定
- 生成器工作台进一步拆分，降低单页复杂度

### P2

只有在前两层完成后，才适合讨论：

- 运行时 DSL
- 在线页面编排
- 跨模块流程编排
- 更面向业务角色的低代码产品形态

---

## 8. 最终判断

最终判断分两句：

1. 现在的 Pantheon 低代码链路，**可以作为内部受控生成器交付**。
2. 现在的 Pantheon 低代码链路，**还不能作为成熟低代码平台产品交付**。

补充判断：

- 以当前主线能力看，`generator + dynamicmodule` 已达到“产品级内部治理工具”标准
- 距离“成熟运行时低代码平台”仍差运行时激活自动化、在线编排与非研发角色可用性三条主线

如果后续对外表达要更准确，建议使用：

- “低代码生成链路”
- “受控模块生成器”
- “业务模块接入加速器”

而不要直接使用一个过满承诺的“低代码平台”名义。
