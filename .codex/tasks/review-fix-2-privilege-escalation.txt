在 pantheon-base（Go modular monolith, Gin+GORM+Casbin）实现"防自提权窄口径"防护。这是安全敏感改动，遵循项目 AGENTS.md 的 L2 边界，不要扩大改动范围。

背景：权限/角色写 API 全部经 Token+Casbin，但 Service 层无纵深校验。若普通角色被误授予指向 /system/permission* 或 /system/role* 的写策略，即可给任意角色（含自身）分配策略，提权到近似 admin。

要求（窄口径）：非 admin 操作者不得创建/修改/删除任何"目标 obj 命中管理端点前缀"的 Casbin 策略。

实现：
1. 在 backend/modules/system/iam/permission/permission_service.go 的 CreatePolicy / UpdatePolicy / DeletePolicy（以及 Import 批量路径）写入 casbin_rule 前，增加校验：
   - 定义受保护前缀集合（常量，便于扩展）：/api/v1/system/permission、/api/v1/system/role。
   - 若待写策略的 path（v1）匹配这些前缀，且当前操作者不是 admin（操作者 roleKeys 从 gin.Context 取，参考现有 common.Get* 或 c.Get("roleKeys")），则拒绝，返回 403 + i18n key "permission.escalation.forbidden"。
   - admin（roleKeys 含 "admin"）不受限。
2. handler 层需把操作者 roleKeys 传入 service（若 service 方法当前无 context 参数，采用与本仓库其他 WithContext 方法一致的注入方式；优先复用已有的从 gin.Context 读取 roleKeys 的 helper，不要新造一套）。
3. 校验逻辑抽成可单测的纯函数，如 isProtectedManagementPolicy(path string) bool 和 canWritePolicy(operatorRoleKeys []string, path string) bool。
4. 补单元测试：admin 可写管理端策略；非 admin 写 /system/permission* 或 /system/role* 被拒；非 admin 写普通业务策略仍允许。
5. 本批不改前端（后端拦截即可）。

约束：不改 Casbin model；不改表结构；不改中间件挂载；只在 Service/Handler 加校验。i18n key 若需要，在 zh/en 资源补上 "permission.escalation.forbidden"。

完成后运行：cd backend && gofmt -w . && go build ./... && go vet ./... && go test ./modules/system/iam/... 并贴出结果。输出改动文件清单与原因。
